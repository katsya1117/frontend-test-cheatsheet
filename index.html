01 — テストとは何か

テストは 「コードが期待通りに動くか確認する別のコード」 です。
書くことは3つだけ。① 状況を作る → ② ユーザー操作をする → ③ 結果を確認する

① 準備（Arrange）
状況を作る
画面をレンダリングして、テストしたい状態にセットアップする
setupWithStore(<UserManage />, {...})
② 操作（Act）
ユーザー操作をする
クリック・文字入力など。操作がないテストはこのステップを省略できる
await user.click(screen.getByText('検索'))
③ 確認（Assert）
結果を確認する
「こうなっているはず」をコードで書く。ここが通らなければテスト失敗
expect(screen.getByText('田中')).toBeInTheDocument()
i
テスト名は日本語で書く。 it('削除後にtoastが表示される', () => {}) のように、「何をテストするか」が一目でわかる名前にする。英語のコードの海に日本語が混じることを気にしなくてOK。
02 — 「何を確認したいか」でシナリオを選ぶ

クリックして開く。コードはフルで掲載しています。そのまま参考にできます。

表示確認 この状態のとき、○○が画面に表示される（またはされない） ›
ポイント： ユーザー操作は不要。preloadedState で「その状態」を作ってレンダリングするだけ。
「存在しない」ことを確認するときは getByText ではなく queryByText を使う。
フルコード例：検索結果なしのメッセージ表示・非表示
// ─── UserManage.test.tsx ─────────────────────────────────────────── import { screen } from '@testing-library/react' import { userSliceReducer } from '@/redux/slices/userSlice' import { autoCompleteSliceReducer } from '@/redux/slices/autoCompleteSlice' import { setupWithStore } from '@test-utils' import UserManage from './UserManage' // storeで使うreducersをまとめる const reducers = { user: userSliceReducer, autoComplete: autoCompleteSliceReducer, } // 「何も変えていない初期状態」をファイル先頭で1回だけ作っておく const baseUserState = userSliceReducer(undefined, { type: '@@INIT' }) const baseAutoCompleteState = autoCompleteSliceReducer(undefined, { type: '@@INIT' }) const basePreloadedState = { user: baseUserState, autoComplete: baseAutoCompleteState, } describe('UserManage', () => { beforeEach(() => jest.clearAllMocks()) it('未検索のとき、検索結果もメッセージも表示されない', () => { // ① 準備：searchCondition が undefined ＝ まだ検索していない状態 setupWithStore(<UserManage />, { reducers, preloadedState: { ...basePreloadedState, user: { ...baseUserState, list: { ...baseUserState.list, searchCondition: undefined, data: undefined, }, }, }, }) // ③ 確認（操作なし） // queryByText → 要素がなければ null を返す（getByText は見つからないと即エラー） expect(screen.queryByText('検索結果')).not.toBeInTheDocument() expect(screen.queryByText('該当するユーザーが見つかりませんでした。')).not.toBeInTheDocument() }) it('検索済みでデータなしのとき、メッセージが表示される', () => { // ① 準備：settingSearched: true ＝ 検索ボタンが押された状態、data: undefined ＝ 結果なし setupWithStore(<UserManage />, { reducers, preloadedState: { ...basePreloadedState, user: { ...baseUserState, searchResultDisp: { ...baseUserState.searchResultDisp, settingSearched: true, }, list: { ...baseUserState.list, searchCondition: {}, data: undefined, }, }, }, }) // ③ 確認 expect(screen.getByText('該当するユーザーが見つかりませんでした。')).toBeInTheDocument() }) })
このコードで覚えること
setupWithStoreReduxのstoreが必要なコンポーネントをレンダリングするヘルパー。@test-utils からimportする。storeとMemoryRouterを自動でラップしてくれる。
preloadedStatestoreの初期状態を自由に指定できる。ベース状態（basePreloadedState）を ... で展開して、変えたい部分だけ上書きする。
queryByText要素が「ない」ことを確認するとき専用。存在しないと null を返す。getByText は存在しないと即テストエラーになるため、.not.toBeInTheDocument() との組み合わせではこちらを使う。
beforeEach(() => jest.clearAllMocks())毎テスト前にmockの呼び出し履歴をリセットする。これがないと前のテストのmock状態が残り、誤検知が起きる。
操作→確認 ボタンを押したら○○が呼ばれる / フォームに入力して送信できる ›
ポイント： setupWithStore が返す user でクリック・入力を再現する。ユーザー操作を含む場合は async/await が必要。
dispatchSpy で「dispatch が呼ばれたか」、getUserList（mock関数）で「正しい引数か」を確認する。
フルコード例：フォーム入力→検索ボタン / クリアボタン

このコードで覚えること
usersetupWithStore が返すオブジェクト。user.type()・user.click() などでユーザー操作を再現する。ユーザー操作は非同期なので必ず await をつける。
dispatchSpysetupWithStore が自動で仕込む。store.dispatch が呼ばれたかどうかを記録している。別途 jest.mock しなくても使える。
getByLabelText<label> のテキストから入力欄を取得する。フォーム要素の取得に最優先で使う（詳しくはセクション06）。
objectContaining引数のオブジェクトが「このキーを含んでいれば OK」と部分一致で確認できる。引数のすべてのフィールドを書く必要がなくなる。
データ表示 storeにデータがあるとき、一覧・リンク・マークが正しく表示される ›
ポイント： preloadedState の list.data.items にモックデータを入れてレンダリング。
ページネーションのモックデータは必ず createMockPagination() を使う（これがないと型エラーや実行エラーになる）。
フルコード例：ユーザー一覧 / 選択リンク / box_user_idマーク

このコードで覚えること

非同期・副作用 画面遷移後に toast が出る / データ取得後に表示が変わる ›
ポイント： 非同期の確認は waitFor または findByText を使う。
Router の state（前の画面から渡すデータ）が必要なときは、setupWithStore ではなく手動でラッパーを作る。
理由：setupWithStore には initialEntries（URLや遷移情報）を渡す機能がない。
フルコード例：削除後遷移でtoastが出る / RouterStateラッパーの書き方
// ─── UserManage.test.tsx ─────────────────────────────────────────── import { toast } from '@/components/ui/sonner' import { userSliceReducer } from '@/redux/slices/userSlice' import { autoCompleteSliceReducer } from '@/redux/slices/autoCompleteSlice' import { configureStore } from '@reduxjs/toolkit' import { Provider } from 'react-redux' import { MemoryRouter } from 'react-router' import { screen, waitFor } from '@testing-library/react' import { setup } from '@test-utils' import UserManage from './UserManage' // toast をmockに差し替える jest.mock('@/components/ui/sonner', () => ({ toast: { success: jest.fn(), error: jest.fn() }, })) const reducers = { user: userSliceReducer, autoComplete: autoCompleteSliceReducer } const baseUserState = userSliceReducer(undefined, { type: '@@INIT' }) const baseAutoCompleteState = autoCompleteSliceReducer(undefined, { type: '@@INIT' }) const basePreloadedState = { user: baseUserState, autoComplete: baseAutoCompleteState } // Routerのstateをセットできるラッパーをヘルパー関数として切り出す // （setupWithStore は initialEntries に対応していないので手動で作る） const setupWithRouterState = (routerState: unknown) => { const store = configureStore({ reducer: reducers, preloadedState: basePreloadedState, }) return setup( <Provider store={store}> <MemoryRouter initialEntries={[{ pathname: '/manage/User', state: routerState, // ← 前の画面から渡したいデータをここにセット }]} > <UserManage /> </MemoryRouter> </Provider> ) } describe('UserManage', () => { beforeEach(() => jest.clearAllMocks()) it('削除完了stateがあるとき toast.success が呼ばれる', async () => { // ① 準備：location.state に deletedUserCd を渡す setupWithRouterState({ deletedUserCd: 'u123' }) // ③ 確認：waitFor でポーリング（非同期なのですぐには toast が呼ばれない） await waitFor(() => { expect(toast.success).toHaveBeenCalledWith('ユーザーu123を削除しました') }) }) it('削除完了stateがないとき toast.success は呼ばれない', async () => { setupWithRouterState(null) // waitFor で少し待ってから「呼ばれていないこと」を確認 await waitFor(() => { expect(toast.success).not.toHaveBeenCalled() }) }) })
このコードで覚えること
waitFor中のコードが成功するまで繰り返し試してくれる（デフォルト最大1秒）。非同期処理の後に確認したいときに使う。
findByTextwaitFor + getByText のショートカット。「○○が表示されるまで待つ」ならこちらが簡潔。例：expect(await screen.findByText('田中')).toBeInTheDocument()
initialEntriesMemoryRouter に画面遷移の履歴を渡す。前の画面から state でデータを受け取るテストで必要。setupWithStore にはこの機能がないため手動でラッパーを作る。
モックコンポーネント操作 Pagination や AutoCompleteMulti を操作したい ›
背景： Pagination と AutoCompleteMulti は全テストで自動的にモックされている（セクション10参照）。内部実装がないため通常の取得方法では操作できず、getByTestId を使う。これはこのプロジェクントの例外的なケース。
フルコード例：Paginationクリック / AutoCompleteMulti選択

03 — どのヘルパーを使うか決める
Reduxのstoreが不要？表示確認・シンプルなUI・storeを使わないコンポーネント
→
setup()
userEvent込みのrender。
返り値：user
↓ storeが必要な場合
通常のRedux + Router？storeのstate・dispatch検証が必要
→
setupWithStore()
store + MemoryRouter付き。
返り値：user, store, dispatchSpy
↓ それでも足りない場合
RouterのstateやURL指定が必要？location.state / initialEntries が必要
→
手動ラッパー
configureStore + MemoryRouter
を自分で組み立てる
各ヘルパーの使い方
// ① setup() — Reduxなし・userEvent付き import { setup } from '@test-utils' const { user } = setup(<MyComponent />) await user.click(screen.getByText('ボタン')) // ② setupWithStore() — Redux + MemoryRouter付き（dispatchSpyも返ってくる） import { setupWithStore } from '@test-utils' const { user, store, dispatchSpy } = setupWithStore(<UserManage />, { reducers: { user: userSliceReducer }, preloadedState: basePreloadedState, }) // ③ 手動ラッパー — RouterのstateやURL指定が必要なとき（シナリオ4参照） import { configureStore } from '@reduxjs/toolkit' import { Provider } from 'react-redux' import { MemoryRouter } from 'react-router' import { setup } from '@test-utils' const store = configureStore({ reducer: reducers, preloadedState: basePreloadedState }) setup( <Provider store={store}> <MemoryRouter initialEntries={[{ pathname: '/manage/User', state: { deletedUserCd: 'u123' } }]}> <UserManage /> </MemoryRouter> </Provider> )
04 — preloadedState の組み立て方

preloadedState は「storeの初期状態」です。「このstateの状態で画面を開いたときに○○か」を確認するためにカスタマイズします。

ファイル先頭に1回だけ書くベース状態（全テストで再利用）
// reducers：このコンポーネントが使うsliceをまとめる const reducers = { user: userSliceReducer, autoComplete: autoCompleteSliceReducer, } // @@INIT を dispatch することで「何も変えていない初期状態」を取得できる const baseUserState = userSliceReducer(undefined, { type: '@@INIT' }) const baseAutoCompleteState = autoCompleteSliceReducer(undefined, { type: '@@INIT' }) const basePreloadedState = { user: baseUserState, autoComplete: baseAutoCompleteState, }
テストごとに「変えたい部分だけ」スプレッドで上書きする
setupWithStore(<UserManage />, { reducers, preloadedState: { ...basePreloadedState, // ← まずベースを展開 user: { ...baseUserState, // ← user も展開（省略するとフィールドが消える） searchResultDisp: { ...baseUserState.searchResultDisp, settingSearched: true, // ← ここだけ変える }, list: { ...baseUserState.list, searchCondition: {}, data: { items: [mockUser], data: [mockUser], pagination: createMockPagination(), // ← pagination は必ずこれを使う }, }, adList: { ...baseUserState.adList, searchCondition: undefined, data: undefined, }, }, }, })
createMockPagination の使い方
// 引数なし → すべてデフォルト値 createMockPagination() // → { current_page: 1, last_page: 1, per_page: 10, from: 1, to: 1, total: 1, ... } // 上書きしたいフィールドだけ渡す → 残りはデフォルト値 createMockPagination({ total: 100, last_page: 10, current_page: 1 }) // → { current_page: 1, last_page: 10, total: 100, ... }
⚠
スプレッドを省略しない： user: { settingSearched: true } のように一部だけ書くと他のフィールドが消えてエラーになる。必ず ...baseUserState で展開してから上書きする。
05 — mock のパターン集

mock は「本物の関数・モジュールを偽物に差し替える」仕組みです。外部への通信やUI副作用をコントロールしてテストを安定させます。

パターン1モジュールの一部だけ差し替え（reducerは本物を使いたい）
jest.requireActual で「実際の実装を残しつつ特定の関数だけmock」できる。reducer などは本物を使いたいが thunk 関数だけ差し替えたいときに使う。
// ファイルの先頭、import の後に書く jest.mock('@/redux/slices/userSlice', () => { // requireActual で本物の実装を取得する const actual = jest.requireActual('@/redux/slices/userSlice') return { ...actual, // reducer / selector など本物をそのまま使う getUserList: jest.fn((arg) => ({ // この関数だけ偽物に差し替える type: 'getUserList', payload: arg, })), } })
パターン2型付きmock — mockした関数をTypeScriptで安全に使う
jest.mock で差し替えた関数は TypeScript の型情報が失われる。ファイル先頭でキャストしておくと .mock.calls などにアクセスできる。
// jest.mock の宣言より後、describe より前に書く const mockedGetUserList = getUserList as unknown as jest.MockedFunction<typeof getUserList> // テスト内での使い方 it('getUserList の引数を直接確認する', async () => { const { user } = setupWithStore(<UserManage />, { reducers, preloadedState: basePreloadedState }) await user.click(screen.getByText('検索')) // mock.calls[0] = 1回目の呼び出し、[0][0] = その第1引数 const firstCallArg = mockedGetUserList.mock.calls[0][0] expect(firstCallArg.auto_complete).toEqual([ { value: 'center1', label: 'center1' }, ]) })
パターン3副作用ライブラリをまるごと差し替え（toast / sonner）
toast などUIライブラリは本物を動かすと実際のUIが必要になる。jest.fn() で空の関数に差し替えて「呼ばれたか・何を渡されたか」だけ確認する。
// ファイル先頭に書く jest.mock('@/components/ui/sonner', () => ({ toast: { success: jest.fn(), error: jest.fn(), }, })) // テスト内での使い方 it('削除完了後に toast.success が呼ばれる', async () => { setupWithRouterState({ deletedUserCd: 'u123' }) await waitFor(() => { expect(toast.success).toHaveBeenCalledWith('ユーザーu123を削除しました') }) }) it('toastは1回だけ呼ばれる', async () => { setupWithRouterState({ deletedUserCd: 'u123' }) await waitFor(() => { expect(toast.success).toHaveBeenCalledTimes(1) }) })
パターン4dispatchSpy — dispatch が呼ばれたか確認する
setupWithStore が自動で store.dispatch に jest.spyOn を仕込んでいる。別途 jest.mock しなくても dispatch 全体を監視できる。
// setupWithStore の返り値から取り出す const { user, dispatchSpy } = setupWithStore(<UserManage />, { reducers, preloadedState: basePreloadedState, }) await user.click(screen.getByText('検索')) // dispatch が呼ばれたかどうか expect(dispatchSpy).toHaveBeenCalled() // dispatch が何回呼ばれたか expect(dispatchSpy).toHaveBeenCalledTimes(1)
パターン5beforeEach で clearAllMocks — mockの汚染を防ぐ
前のテストのmock呼び出し履歴が次のテストに残ってしまう。describe の中に必ず書く。
describe('UserManage', () => { // ← describe の直下に書く。beforeEach = 各テストの前に実行される beforeEach(() => jest.clearAllMocks()) it('テスト1', () => { ... }) it('テスト2', () => { ... }) // ← テスト1のmock履歴はここで消えている })
06 — 要素の取得方法（クエリ）— この順番で選ぶ

「どうやって要素を取得するか」にはルールがあります。アクセシビリティに優れた方法を優先することで、実際のユーザーに近い視点でテストが書けます。

1位getByRole()button, link, heading, textbox, checkbox などARIAロール
2位getByLabelText()フォームの <label> と紐づいた入力欄
3位getByText()テキスト内容・ボタンのラベル・メッセージ
最後getByTestId()モックコンポーネントの操作のみ（このプロジェクト限定の例外）
// ✅ 1位：ロールで取得（button, link, heading, textbox, checkbox...） screen.getByRole('button', { name: '検索' }) // name でテキストを絞り込める screen.getByRole('link', { name: '選択' }) // <a> タグを取得 screen.getByRole('heading', { name: 'ユーザー管理' }) screen.getByRole('checkbox') // ✅ 2位：ラベルで取得（フォーム要素に最優先で使う） screen.getByLabelText('表示名') // <label>表示名</label> と紐づいた input screen.getByLabelText('メールアドレス') // ✅ 3位：テキストで取得（ボタン・メッセージなど） screen.getByText('検索') screen.getByText('該当するユーザーが見つかりませんでした。') // ⚠ 最後の手段（このプロジェクトではモックコンポーネントの操作時のみ使う） screen.getByTestId('pagination-mock') screen.getByTestId('auto-complete-multi') // ─── get / query / find の違い ─────────────────────────────────── // getBy → 見つからないとその場でテストエラー（存在することが前提のとき） screen.getByText('田中') // queryBy → 見つからないと null を返す（存在しないことを確認したいとき） screen.queryByText('検索結果') // → null expect(screen.queryByText('検索結果')).not.toBeInTheDocument() // findBy → 非同期で表示される要素を待って取得（waitFor + getBy のショートカット） await screen.findByText('田中') // → 表示されるまで待ってから取得

08 — TypeScript テストのポイント
MockedFunctionmockした関数をTypeScriptで安全に扱う
jest.mock で差し替えた関数は TypeScript の型情報が失われる。as unknown as jest.MockedFunction でキャストすると .mock.calls などが型安全に使える。
// ① jest.mock でモジュールを差し替える jest.mock('@/redux/slices/userSlice', () => { const actual = jest.requireActual('@/redux/slices/userSlice') return { ...actual, getUserList: jest.fn((arg) => ({ type: 'getUserList', payload: arg })) } }) // ② ファイル先頭で型付きmockとして宣言する const mockedGetUserList = getUserList as unknown as jest.MockedFunction<typeof getUserList> // ③ テスト内で使う it('getUserList の第1引数を確認する', async () => { const { user } = setupWithStore(<UserManage />, { reducers, preloadedState: basePreloadedState }) await user.click(screen.getByText('検索')) // mock.calls[0][0] → 1回目の呼び出しの第1引数 const firstCallArg = mockedGetUserList.mock.calls[0][0] expect(firstCallArg.auto_complete).toEqual([ { value: 'center1', label: 'center1' }, ]) })
as unknown as 型名テスト用モックデータを作るときの型変換
テストではすべてのフィールドを持つオブジェクトを作るのが大変。必要なフィールドだけ持つオブジェクトを as unknown as 型名 で強制変換する。
// すべてのフィールドを書かなくてよい const mockUser = { user_cd: 'u001', disp_name: '田中', email: 'tanaka@example.com', } as unknown as UserInfo // box_user_id も追加する場合 const mockUserWithBox = { user_cd: 'u002', disp_name: '鈴木', email: 'suzuki@example.com', box_user_id: 'box-1', } as unknown as UserInfo

型に合わない値を書いたときにコンパイルエラーにしたい場合は satisfies を使う。as unknown as より厳密。
// satisfies でモックデータを型安全に作る const mockPagination = { current_page: 1, last_page: 5, per_page: 10, total: 50, } satisfies Partial<Pagination> // jest.fn() に型をつける const onClick: jest.Mock = jest.fn() render(<Button onClick={onClick}>送信</Button>) expect(onClick).toHaveBeenCalledOnce()

10 — グローバルmock済みコンポーネント一覧

「モック済み」とは： test/jest.setup.ts に設定があり、すべてのテストファイルで自動的にこれらが「空っぽの偽物」に置き換えられています。テスト対象のコンポーネントを余計な依存なしに動かせる反面、内部実装は動きません。

モック済みコンポーネント操作するとき備考
Pagination getByTestId('pagination-mock') をクリック クリックするとページ遷移イベントが発火する。通常の getByRole では取得できない
AutoCompleteMulti getByTestId('auto-complete-multi') に user.selectOptions() 内部的に <select> 要素として扱われる
react-router 通常は意識しなくてOK useNavigate / useLocation は自動でmockされる。Router の state が必要なときだけ MemoryRouter を手動で書く（シナリオ4参照）
@/components/ui/tabs 操作しない テストで意識しなくてOK
@/components/frame/Layout 操作しない テストで意識しなくてOK
i
Pagination と AutoCompleteMulti は内部実装がないため、getByRole / getByText では取得できません。このプロジェクントではこの2つだけ例外的に getByTestId を使います。

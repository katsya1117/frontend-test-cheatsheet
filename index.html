02 — 単体テスト（関数・ロジック）
UNIT パターン選択フロー
値を返す関数？引数 → 戻り値
→
expect(fn(x)).toBe(y)
↓
エラーを投げる？バリデーション等
→
expect(() => fn()).toThrow()
↓
オブジェクト・配列？構造を検証
→
toEqual / toMatchObject
↓
複数ケースを網羅？境界値・異常値
→
test.each([...cases])
// 例: formatPrice(1000) → "¥1,000" describe('formatPrice', () => { it('正常値を整形する', () => { expect(formatPrice(1000)).toBe('¥1,000') }) it('負の値でエラーを投げる', () => { expect(() => formatPrice(-1)).toThrow('Invalid amount') }) test.each([ [0, '¥0'], [999, '¥999'], [10000,'¥10,000'], ])('formatPrice(%i) → %s', (input, expected) => { expect(formatPrice(input)).toBe(expected) }) })
03 — UIコンポーネントテスト
UI パターン選択フロー
表示確認？テキスト・要素の存在
→
getByText / getByRole
↓
クリック・入力？ユーザー操作を再現
→
userEvent.click / type
↓
条件付き表示？props / stateで変化
→
rerender / queryByText
↓
存在しないことを確認？非表示・削除後
→
not.toBeInTheDocument()
// 例: Buttonコンポーネント import { render, screen } from '@testing-library/react' import userEvent from '@testing-library/user-event' it('クリックでonClickが呼ばれる', async () => { const onClick = jest.fn() render(<Button onClick={onClick}>送信</Button>) await userEvent.click(screen.getByRole('button')) expect(onClick).toHaveBeenCalledOnce() }) it('disabled時はクリックできない', async () => { const onClick = jest.fn() render(<Button disabled onClick={onClick}>送信</Button>) await userEvent.click(screen.getByRole('button')) expect(onClick).not.toHaveBeenCalled() })
04 — 非同期・APIテスト
ASYNC パターン選択フロー
APIを呼ぶ関数？fetch / axios
→
jest.mock でモック化
↓
成功時の表示確認？データ取得後のUI
→
await findByText()
↓
エラー時の表示確認？失敗ハンドリング
→
mockRejectedValue → findByText
↓
ローディング状態？取得中のUI
→
getByText('読み込み中')
// 例: データ取得コンポーネント jest.mock('../api') it('データ取得後に一覧表示', async () => { (api.getUsers as jest.Mock).mockResolvedValue([ { id: 1, name: '田中' } ]) render(<UserList />) expect(await screen.findByText('田中')).toBeInTheDocument() }) it('失敗時にエラーメッセージ表示', async () => { (api.getUsers as jest.Mock).mockRejectedValue(new Error('Network error')) render(<UserList />) expect(await screen.findByText('エラーが発生しました')).toBeInTheDocument() })
05 — クエリの優先順位（Testing Library）
アクセシビリティに優れた順に使う
1st
getByRole button, link, heading, textbox, checkbox — ARIAロール
2nd
getByLabelText フォームラベルと関連付けられた要素
3rd
getByText / getByPlaceholderText テキスト内容・プレースホルダー
最後
getByTestId 実装の詳細に依存するため、他の手段がない場合のみ使用
// ✅ Good — ロールで取得 screen.getByRole('button', { name: '送信' }) // ✅ Good — ラベルで取得 screen.getByLabelText('メールアドレス') // ❌ Avoid — 実装詳細に依存 screen.getByTestId('submit-btn')
06 — TypeScript テストのポイント
TS 型安全なテストの書き方
Props の型を担保型エラーをコンパイル時に検出
// 型定義をそのまま使う const props: ButtonProps = { label: '送信', onClick: vi.fn(), disabled: false, }
satisfies でモックデータ型を絞って誤入力防止
// satisfiesで型安全に const mockUser = { id: 1, name: '田中', email: 't@example.com' } satisfies User
// jest.Mockで型付きモック関数 const mockFetch = jest.fn() as jest.MockedFunction<typeof fetch> mockFetch.mockResolvedValue({ json: () => Promise.resolve({ id: 1 }), ok: true, } as Response)
